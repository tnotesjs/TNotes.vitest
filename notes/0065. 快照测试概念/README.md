## 1. 本节内容

- 快照测试将输出序列化保存
- 适用场景（UI 渲染/API 响应）
- 优点（快速验证变化）
- 缺点（维护成本高）
- 最佳实践
- 快照 vs 具体断言

## 2. 评价

快照测试适合捕获复杂数据结构的变化，降低测试维护成本。但快照可能变得过大难以审查，开发者可能盲目更新，需要团队规范。

## 3. 什么是快照测试

快照测试将函数或组件的输出序列化后保存到文件，后续运行时自动对比。如果输出发生变化，测试失败。

```typescript
test('snapshot', () => {
  const result = {
    name: 'vitest',
    version: 4,
  }
  expect(result).toMatchSnapshot()
})
```

## 4. 适用场景

- UI 组件的渲染输出
- API 响应结构
- 复杂数据结构
- 配置对象
- 错误消息

## 5. 优点

- 快速验证输出是否变化
- 不需要手动编写断言
- 适合捕获意外的变化
- 降低测试维护成本

## 6. 缺点

- 快照可能变得过大难以审查
- 开发者可能盲目更新快照
- 快照文件需要提交到版本控制
- 团队需要审查快照变更

## 7. 维护成本

快照测试需要定期维护：

- 代码变更时需要更新快照
- 大型快照难以理解和审查
- 快照可能包含不必要的信息

## 8. 最佳实践

- 保持快照小而专注
- 只快照关键的输出
- 定期清理无用的快照
- 在 PR 中仔细审查快照变更

## 9. 快照 vs 具体断言

| 维度     | 快照测试               | 具体断言             |
| -------- | ---------------------- | -------------------- |
| 维护成本 | 低（自动更新）         | 高（手动维护）       |
| 可读性   | 中（需要查看快照文件） | 高（断言直接可见）   |
| 精确度   | 低（捕获所有变化）     | 高（只检查指定内容） |
| 适用场景 | 复杂输出               | 关键逻辑             |

## 10. 选择建议

- 输出结构复杂且稳定：使用快照
- 关键业务逻辑：使用具体断言
- UI 组件渲染：可以使用快照
- API 响应：可以使用快照
